跟着综述了解向量数据库

本文是综述Survey of Vector Database Management Systems的阅读笔记

原文arxiv

前置知识：矢量数据库与传统数据库简介

矢量数据库:

  • 处理高维数据：矢量数据库旨在管理和存储高维空间中的数据。 这对于像这样的应用程序特别有用 机器学习，其中数据点（例如图像或文本）可以表示为多维空间中的向量。
  • 针对相似性搜索进行了优化：矢量数据库的一个突出特点是它们执行相似性搜索的能力。 这些数据库不是基于精确匹配来查询数据，而是允许用户检索与给定查询“相似”的数据，这使得它们对于图像或文本检索等任务非常有价值。
  • 可扩展用于大型数据集：随着人工智能和机器学习应用程序的不断增长，它们处理的数据量也在不断增长。 矢量数据库是按比例构建的，确保它们可以处理大量数据而不影响性能。

传统数据库:

  • 结构化数据存储：传统数据库，如关系数据库，旨在存储结构化数据。 这意味着数据被组织到预定义的表、行和列中，确保数据的完整性和一致性。
  • 针对 CRUD 操作进行了优化：传统数据库主要针对CRUD操作进行优化。 这意味着它们旨在高效地创建、读取、更新和删除数据条目，使其适用于从 Web 服务到企业软件的各种应用程序。
  • 固定架构：许多传统数据库的定义特征之一是其固定模式。 一旦定义了数据库结构，进行更改可能会非常复杂且耗时。 这种刚性可确保数据一致性，但灵活性可能不如某些现代数据库的无模式或动态模式性质。

☆ 传统数据库经常与嵌入的复杂性作斗争，而矢量数据库很容易解决这一挑战。

★ comment: 向量数据库区别于传统数据库的关键就在于向量数据库使用向量来衡量“相似性”。

目前面临的障碍

the vagueness of semantic similarity, large size of vectors, high cost of similarity comparison, lack of a natural partitioning that can be used for indexing, difficulty of efficiently answering “hybrid” queries that require both attributes and vectors.

  1. 语义相似性的模糊性：指如何计算语义的相似性
  2. 向量的大尺寸：在特定场景下有跨多个数据page的向量大小
  3. 相似性比较的高成本：计算复杂度高
  4. 缺乏可用于索引的自然分区：向量没有自然的顺序，如何进行index
  5. 难以高效地回答需要属性和向量的“混合”查询：涉及多个向量如何查询

Distance - 距离即相似性

★ 距离，用于度量两个向量之间的相似性

  1. hamming：直接用向量的各维度的值相等的个数来作为距离

  2. inner product：面临的问题是大向量的点积天生更长

  3. cosine：使用向量夹角角度作为距离

  4. minkowski：闵可夫斯基距离就是欧几里得距离的推广。其极限是切比雪夫距离，切比雪夫距离取两向量间相差最大的维度的差为距离。

  5. mahalanobis：
    考虑了各维度分布的方差，进行标准化，以消除不同方面量纲不同带来的影响

    • 补充：了解minkowski、mahalanobis

★ 这篇综述认为，没有定论哪个更好，目前。minkowski距离等方案就是为解决维度诅咒等问题提出的方案。
☆ comment：这个维度诅咒究竟如何理解，是否真的是一种“诅咒”？笔者是未理解的。

Query - 查询

query的操作：一个查询向量（如何产生？）在向量数据库中进行查询（寻找向量集合的某个子集）

★ 基础操作：

  1. range search：按照半径去搜索，参数r。
  2. k-近邻：k个最近的邻居，参数k。
  3. ann：相似度（距离）为（接近）c的k个邻居，参数c和k。

★ 扩展

  1. 批量查询：
    文章讨论了硬件加速等问题。

  2. 多向量：
    被查询实体和查询query均可以由多个向量表示。

    ☆目前有单向量实体多向量query和多向量实体单向量query

    ★ 但是还没有多向量实体多向量query的产品出来

Index - 索引

★ 索引面临的一个问题：分布可变性

数据的分布不确定，可变，导致起索引通常需要在有增删改时重构索引。

☆ 下面是文中所述的分区和存储的思路，笔者大概确乎是理解的不特别到位。

分区

  1. 随机化：利用（可能重构）各个独立事件（维度）的概率分布区分真正相似和不相似的向量。
  2. 可学习分区：监督/无监督的学习，研究出向量的内在结构以进行分区。
  3. 导航分区：导航索引，不是绝对的分区。（？）

存储

  1. 量化：将向量投影到一个利用起来更高效的向量空间中（通常维度更小），以使用更少的存储空间并努力减小这一过程的信息损失。

    ☆ 量化的比LSH更能节约空间。

  2. 磁盘驻留设计：减少检索时间。

Index - 三类索引

☆ 这部分算是一个核心内容，包括基于表、基于树和基于图的方法。

Table based

  • 基于表的方法的核心思路：将向量划分为多个桶（bucket），使桶内向量之间相似。

LSH（局部敏感散列）

  1. 将高维向量划分为多个局部区域（临近区域），对每个局部区域使用不同的哈希函数进行哈希。
    • comment：这里的划分未局部区域，相当于给向量分段。
  2. 哈希函数的设计使得在高维空间中距离较近的向量在局部区域内有很大的概率被映射到相同的哈希桶中。
  3. 当进行相似性搜索时，将查询向量通过相同的哈希函数映射到相应的局部区域，并在对应的局部区域内搜索相似的向量。

★ 可学习哈希：

可能使用k-means 等机器学习方法来划分hash簇，以k均值各个分类的中心点作为hash键.

  • 需注意，此类方法可能对数据的分布外增加敏感，导致错误，在现有系统中支持也不多。

table based 的方法的量化：

把向量分成m段，每段分布作分区和索引。

比如预备分成k类，每段就固定分成k/m类。

计算相似性的复杂度大幅降低：
Flat indexex IVFSQ IVFADC …

Tree based

  • 基于树的方法的核心思路：对基于表的方法的嵌套。

基于树的索引

对于基于树的索引，主要考虑的是将S递归地拆分为搜索树的拆分策略。

一种自然的方法是基于距离进行拆分。主要技术包括基于支点的树，例如VP-tree和M-tree，k-means树和基于深度学习的树。

虽然这些树对于低维度数据有效，但在应用于高维度数据时，它们会受到维度诅咒的影响。

“高维度基于树的索引倾向于依赖随机化来执行节点拆分。特别地，“快速近似最近邻库”（FLANN）将随机化与通过主成分分析（PCA）学习的分区相结合，扩展了PKD-tree技术，“近似最近邻ohyeah”（ANNOY）类似于随机投影树（RPTree）。”

☆ KD-tree：
Principal Component Trees、Random Projection Trees。
思路就是每个节点可能按照某个维度的值生出两个子节点

☆ Ball-tree：
球树是一种二叉树，其中每个节点定义一个包含要搜索的点的子集的 D 维球。树的每个内部节点将数据点划分为与不同球相关联的两个不相交的集合。虽然球本身可能相交，但每个点都会根据其距球中心的距离分配给分区中的一个或另一个球。树中的每个叶节点定义一个球并枚举该球内的所有数据点。

☆ 补充
倒排索引：类似于关键词索引（算树吗？还是表？）

  • 倒排索引就是像统计关键词然后用关键词搜索文章的思路。

Graph based

  • 基于图的方法的核心思路：连接虚拟的边，将向量数据作为节点连接成 图 这种数据结构来进行相似性搜索。

由于图的构造，导致搜索问题可以被视作图上的搜索问题，节点间的相似性等问题可以用图中的距离来度量。

★ comment：笔者认为，甚至可以用异质图吧。

  1. k近邻图KNNG： 这种图中每个节点与他最近的k个节点相连
  2. MSN单调搜索网络Monotonic Search Networks：KNNG不一定是连通的
  3. HNSW层次可导航小世界：
    分层的可导航小世界（Hierarchical Navigable Small World，HNSW）是一种基于图的数据结构，它将节点划分成不同层级，贪婪地遍历来自上层的元素，直到达到局部最小值，然后切换到下一层，以上一层中的局部最小值作为新元素重新开始遍历，直到遍历完最低一层。
